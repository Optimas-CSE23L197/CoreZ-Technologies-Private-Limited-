Section 02

How Data Flows in an Emergency

Here is exactly what happens, step by step, from the moment a QR card is scanned:

1
QR Code Scanned

Someone scans the student's card using any standard smartphone camera no app installation required for the scanner.

2
Secure HTTPS Page Loads

An encrypted web page loads instantly, displaying the student's name, photo, and medical info over a secure TLS connection.

3
Scan Event Securely Logged

The scan timestamp and device location (if permitted by the scanner's device) are encrypted and stored securely in the database.

4
Parent Notified Instantly

A real-time push notification is sent to the registered parent's Guardian app, including time and approximate location of the scan.

5
Parent Views Location on Map

The parent can view the scan location pinned on a map within their app to quickly navigate to their child's location.

 The parent's phone number is never displayed on the scan page. The emergency call button initiates a direct call without revealing the number to the scanner.

Data Storage

  • All data is stored on secured cloud servers with strict role-based access controls and full audit logging
  • Student profile data is stored in an AES-256 encrypted database
  • Scan logs and location data are stored in a separate partition with additional access restrictions
  • Encrypted backups are taken automatically and stored in geographically separated storage
  • All data is stored in India-based or India-compliant data centres in accordance with applicable law

Security Practices

  • HTTPS everywhere Every page, API call, and webhook uses TLS 1.2+ encryption
  • Password hashing All passwords are hashed using bcrypt or argon2. We never store plain-text passwords under any circumstances
  • Session management Login sessions expire automatically after a period of inactivity
  • Input validation & sanitisation All user inputs are strictly validated to prevent SQL injection, XSS, and other common attacks
  • Rate limiting All API endpoints are rate-limited to prevent abuse, brute-force, and scraping attempts
  • Role-based access control (RBAC) Staff, schools, and parents can only access data they are explicitly permitted to see
  • Security reviews We conduct periodic security assessments and penetration testing of the platform
Section 06
6

Access Controls

  • Only a small number of authorised RESQID technical staff can access backend systems, with every access event logged with timestamp and reason
  • School administrators can only view profiles of students explicitly enrolled under their own institution
  • Parents can only view their own child's data, scan logs, and location history
  • No third party including advertisers, analytics providers, or partners has standing access to student data
Section 07
7

Data Breach Response

In the unlikely event of a data security incident, RESQID commits to the following response protocol:

  • Notify all affected users within 72 hours of becoming aware of the breach
  • Report to relevant Indian data protection authorities as required under the DPDPA 2023
  • Immediately contain, investigate, and remediate the security issue with full priority
  • Provide clear, plain-language guidance to affected users on steps they should take to protect themselves
  • Publish a post-incident report for transparency after the incident is resolved

To report a vulnerability or security concern: security@getresqid.in
